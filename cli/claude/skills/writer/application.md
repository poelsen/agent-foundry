# Application, cover letter, personal statement, grant

Register: professional, warm, genuinely motivated. Generic enthusiasm ("I am thrilled to apply for this exciting opportunity") is the failure mode; specific, evidenced motivation is the goal.

## Require the target document first

Ask for the job posting, call for proposals, or funder's criteria before drafting; fold this into the single intake batch from SKILL.md step 1 rather than adding a second round of questions. An application written blind is generic by construction. Pull **three to five stated requirements or evaluation criteria** from it and answer each.

## Structure

- **One STAR paragraph per key requirement:** Situation → Task → Action → **quantified Result**. The number is what makes impact legible ("cut release time from 6 weeks to 9 days"), and legibility is why STAR is the default here, not because it's a proven callback trick. Commit to it; don't over-claim it.
- **Mirror one or two key phrases** from the posting naturally, so the reader sees the fit. Do not keyword-stuff, mechanical keyword-mirroring is a weak and decaying signal, and it reads as gaming.
- **Lead with the problem and the outcome, not the applicant's prestige.** For a grant, align explicitly to the funder's evaluation criteria and state the concrete ask: amount, duration, deliverables.

## The hard line

**Never fabricate an achievement, a number, or a credential.** If you don't have the real figure behind a result, ask the user for it or write the result qualitatively, do not invent a plausible one. In applications this is not just an AI-tell issue; it's a misrepresentation risk that can cost the user the role or the grant.

When writing as the author, read `voice-profile.md` for first-person motivation; read `exemplars.md` for how to make evidence legible without inflating it.
